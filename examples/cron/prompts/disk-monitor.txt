You are performing a disk usage health check. Work through each step. Only send an alert message if a threshold is exceeded or a retention anomaly is found.

---

STEP 1 — Disk Usage Check
Run `df -h` and check usage for the following mount points and paths:
- / (root filesystem)
- /var/opt/gitlab/backups     [CUSTOMIZE: GitLab backup path]
- /var/opt/gitlab/git-data    [CUSTOMIZE: GitLab repository storage]
- /var/log/journal             [CUSTOMIZE: systemd journal logs]
- /var/lib/docker              [CUSTOMIZE: Docker storage]
- [CUSTOMIZE: add additional paths]

For each path, report: path, total size, used, available, usage percentage.

Alert threshold: 85% [CUSTOMIZE: change threshold]
If any path exceeds the threshold, proceed to Step 2.
If all paths are below the threshold, output only: "✅ All disks healthy — [timestamp]" and stop.

STEP 2 — Root Cause Identification (only if threshold exceeded)
For each path that exceeded the threshold:
Run `du -sh <path>/*` (one level deep) to identify the largest sub-directories.
List the top 5 largest items with sizes.
Categorize each as: backup files / log files / Docker layers / application data / unknown.

STEP 3 — Cleanup Plan (only if threshold exceeded)
Based on the findings in Step 2, produce a cleanup plan:

For backup files:
- List backups older than 30 days [CUSTOMIZE: retention period]
- Estimate reclaim size
- Provide exact commands to remove safely

For Docker:
- Run `docker system df` to show reclaimable space
- Provide: `docker system prune --volumes -f` with estimated reclaim size
- List stopped containers, dangling images, and unused volumes separately

For journal logs:
- Provide: `journalctl --disk-usage`
- Provide: `journalctl --vacuum-time=14d` with estimated reclaim [CUSTOMIZE: retention]

For application data:
- Flag for manual review — do not suggest automatic deletion

STEP 4 — Retention Check
Check Prometheus TSDB retention:
- Query: `prometheus_tsdb_storage_blocks_bytes` at [CUSTOMIZE: http://prometheus.internal:9090]
- Expected retention: [CUSTOMIZE: 15d] — flag if blocks older than retention period exist

Check Loki chunk retention:
- Query Loki status at [CUSTOMIZE: http://loki.internal:3100/loki/api/v1/status/buildinfo]
- Flag if chunk storage is growing faster than expected (>10% increase since last check)

OUTPUT CONTRACT (REQUIRED):
Mode A — Healthy state:
- Return exactly one line:
  `✅ All disks healthy — <timestamp ISO-8601 UTC>`
- No extra lines, tables, or commentary.

Mode B — Alert state (if any path >= threshold OR retention anomaly):
- Return Markdown with this exact section order:
  1. `# 🚨 DISK ALERT`
  2. `## Summary`
  3. `## Disk Usage`
  4. `## Top Consumers`
  5. `## Cleanup Plan`
  6. `## Retention Status`
  7. `## Execution Timestamp`

- `## Summary`:
  - Bullet list with impacted paths only.
  - Format per bullet: `- <path> | <usage>% used | <available> free | Status: WARNING|CRITICAL`

- `## Disk Usage`:
  - Markdown table with exact columns:
    `Path | Total | Used | Available | Usage% | Status`

- `## Top Consumers`:
  - For each impacted path, list top 5 items from `du -sh <path>/*`.
  - Format: `### <path>` followed by 5 bullets: `- <size> <item> (<category>)`

- `## Cleanup Plan`:
  - Numbered list with exact commands in backticks.
  - Each step must include `Risk:` and `Estimated reclaim:`.

- `## Retention Status`:
  - Exactly two lines:
    - `Prometheus: OK|WARNING — <reason>`
    - `Loki: OK|WARNING — <reason>`

- `## Execution Timestamp`:
  - Exactly one line:
    - `<timestamp ISO-8601 UTC>`
